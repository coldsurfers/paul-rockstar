# RSS와 REST API를 하나의 피드로 — React Native에서 이기종 콘텐츠를 통합한 이야기

> "데이터 소스가 다르다는 건 사용자가 알 필요 없는 사실이다."

---

앱에 콘텐츠 섹션을 만들고 있었다. 기획 의도는 단순했다 — 사용자가 관심 가질 만한 글을 카테고리별로 보여주고, "전체" 탭에서는 모든 카테고리의 글을 최신순으로 섞어서 보여주는 것.

문제는 데이터 소스가 하나가 아니었다는 점이다.

- 외부 블로그 2개: RSS 피드로만 접근 가능
- 자체 뉴스룸 2개 카테고리: WordPress REST API로 접근

카테고리별 탭은 단순하다. 각 소스에 맞는 훅을 호출하고, 해당 데이터를 렌더링하면 끝이다. 하지만 "전체" 탭은 다르다. 서로 다른 형태의 데이터를 하나의 타임라인으로 합쳐야 한다.

---

## 첫 번째 문제: 타입이 다르다

RSS 아이템과 WordPress 포스트는 구조가 완전히 다르다. 제목 하나만 봐도 RSS는 `item.title`이고, WordPress는 `item.title.rendered`다. 썸네일도 RSS는 본문 HTML에서 `<img>` 태그를 파싱해야 하고, WordPress는 전용 필드가 있다. 링크, 날짜, 카테고리 — 전부 접근 경로가 다르다.

이걸 하나의 리스트에서 렌더링하려면 통합 타입이 필요하다.

```typescript
type UnifiedArticleItem =
  | { type: 'rss'; data: FeedItem; category: Category; date: Date }
  | { type: 'wp'; data: CMSPost; category: Category; date: Date };
```

Discriminated union으로 모델링했다. `type` 필드로 분기하면 TypeScript가 `data`의 타입을 자동으로 좁혀준다. `renderItem`에서 `item.type === 'rss'`로 분기하는 순간, `item.data`는 `FeedItem`으로 추론된다.

이렇게 하면 **데이터 소스의 차이를 타입 시스템이 관리**한다. 런타임에서 `if-else`로 분기하되, 각 분기 안에서는 타입이 보장되니까 실수할 여지가 줄어든다.

---

## 두 번째 문제: 로딩 타이밍이 다르다

RSS는 한 번 요청하면 전체 피드가 온다. 페이지네이션이 없다. 반면 WordPress API는 infinite query로 페이지 단위로 로드한다. 두 소스의 데이터가 도착하는 시점이 다르고, WordPress는 사용자가 스크롤할 때마다 새 페이지가 추가된다.

이걸 하나의 정렬된 리스트로 만들려면, "지금까지 도착한 데이터" 전체를 합치고 날짜순으로 정렬해야 한다. 그런데 WordPress에서 새 페이지가 로드될 때마다 전체를 다시 정렬하면 이미 보고 있던 아이템의 순서가 바뀔 수 있다. 사용자가 3페이지를 보고 있는데 갑자기 아이템 순서가 셔플되면 곤란하다.

해결 방법은 **incremental merge**다.

```typescript
// 개념적 흐름
setSortedData(prev => {
  const newItems = [];

  rssData?.forEach(item => {
    if (!prev에_이미_있는지_확인(item)) {
      newItems.push({ type: 'rss', data: item, date: new Date(item.pubDate) });
    }
  });

  wpData?.forEach(post => {
    if (!prev에_이미_있는지_확인(post)) {
      newItems.push({ type: 'wp', data: post, date: new Date(post.date) });
    }
  });

  // 기존 데이터는 순서 유지, 새 데이터만 정렬해서 뒤에 붙인다
  return prev.concat(newItems.sort((a, b) => b.date - a.date));
});
```

기존 아이템은 그대로 두고, 새로 도착한 아이템만 정렬해서 뒤에 붙인다. 이렇게 하면 사용자가 보고 있는 리스트의 앞부분은 절대 흔들리지 않는다.

## 세 번째 문제: 캐시를 공유해야 한다

"전체" 탭과 개별 카테고리 탭은 같은 데이터를 다른 방식으로 보여준다. "전체" 탭에서 이미 로드한 WordPress 포스트를 "스토리" 탭으로 이동했을 때 다시 요청하면 낭비다.

해법은 의외로 단순했다. **"전체" 탭에서도 개별 카테고리와 동일한 query key를 사용**하는 것이다.

```typescript
// "스토리" 탭
usePosts({ categories: [CATEGORY_A_ID] });

// "전체" 탭 — 동일한 query key로 호출
usePosts({ categories: [CATEGORY_A_ID] });
usePosts({ categories: [CATEGORY_B_ID] });
```

TanStack Query는 query key가 같으면 캐시를 공유한다. "전체" 탭에서 먼저 로드하든, 개별 탭에서 먼저 로드하든, 한번 받아온 데이터는 다른 탭에서 즉시 사용된다. 별도의 캐시 동기화 로직이 필요 없다.

`staleTime`을 1시간으로 설정해서, 한번 로드된 콘텐츠는 1시간 동안 재요청하지 않는다. 콘텐츠 특성상 실시간성이 중요하지 않으니까.

## 네 번째 문제: Infinite Query가 두 개다

"전체" 탭에서는 WordPress 쿼리가 2개 동시에 돌아간다. 각각 `hasNextPage`, `fetchNextPage`, `isFetchingNextPage`를 가지고 있다. 이걸 하나의 리스트 컴포넌트에 연결하려면 병합이 필요하다.

```typescript
const hasNextPage = hasNextPageA || hasNextPageB;

const fetchNextPage = async () => {
  const promises = [];
  if (hasNextPageA && !isFetchingA) promises.push(fetchNextPageA());
  if (hasNextPageB && !isFetchingB) promises.push(fetchNextPageB());
  await Promise.all(promises);
};
```

둘 중 하나라도 다음 페이지가 있으면 "더 있음"이고, 각각 독립적으로 다음 페이지를 가져온다. `Promise.all`로 병렬 실행하면 두 소스의 다음 페이지가 동시에 도착한다.

## 다섯 번째 문제: 페이지네이션 UI

리스트 컴포넌트에 한 가지 설계 결정이 더 있었다. 서버 페이지네이션과 별개로, **클라이언트 쪽 페이지네이션**을 한 겹 더 얹었다.

데이터가 수십 개씩 한 화면에 쏟아지면 사용자가 압도된다. 그래서 4개씩 끊어서 horizontal paging으로 보여주고, 이전/다음 버튼으로 넘기게 했다. 핵심은 **사용자가 마지막 클라이언트 페이지에 도달했을 때, 서버에서 다음 배치를 자동으로 prefetch**하는 것이다.

```typescript
useEffect(() => {
  const isLastPage = currentPage === totalPage && totalPage > 0;
  if (hasNextPage && isLastPage && !isFetchingNextPage) {
    fetchNextPage();
  }
}, [currentPage, totalPage, hasNextPage, isFetchingNextPage]);
```

사용자는 "다음" 버튼을 누를 뿐이지만, 뒤에서는 서버 데이터가 미리 준비되고 있다. 클라이언트 페이지네이션이 서버 페이지네이션의 트리거 역할을 하는 2-tier 구조다.

---

## 한 발짝 물러서 보면

이 작업의 본질은 **"사용자에게는 하나의 리스트인데, 뒤에서는 네 개의 소스가 돌아간다"**는 복잡성을 감추는 것이었다.

정리하면 이런 레이어들이 쌓여 있다.

| 계층 | 역할 |
|---|---|
| 데이터 소스 | RSS 2개 + WordPress API 2개 |
| 통합 타입 | Discriminated union으로 이기종 데이터 단일화 |
| 캐시 | 동일 query key로 탭 간 캐시 공유 |
| 병합 | Incremental merge로 기존 순서 보존 |
| 서버 페이지네이션 | 2개의 infinite query를 Promise.all로 병합 |
| 클라이언트 페이지네이션 | 4개씩 horizontal paging + 자동 prefetch |

각 레이어는 단독으로 보면 복잡하지 않다. 하지만 이 레이어들이 서로 맞물려 돌아가게 만드는 것이 이 피처의 핵심이었다.

## 회고: 지금 다시 만든다면

완성하고 나서 몇 가지 아쉬운 점이 보였다.

**incremental merge의 부작용.** 기존 아이템 순서를 보존하기 위해 새 아이템을 뒤에 붙이는 방식을 택했는데, 이러면 전체 리스트가 완벽한 날짜순이 아닐 수 있다. 새로 도착한 아이템 중 기존 아이템보다 최신인 것이 뒤에 붙는 경우가 생긴다. `useMemo`로 매번 전체를 정렬하되, 리스트 key를 안정적으로 유지하는 방식이 더 나았을 수 있다.

**중첩 FlatList의 성능 리스크.** 현재 구조는 FlatList가 3중으로 중첩되어 있다 — outer scroll, horizontal paging, vertical item list. React Native의 FlatList는 중첩에 취약하다. `FlashList`로의 교체나, 단일 리스트 + section 패턴으로의 리팩토링을 고려하고 있다.

**상태 관리의 복잡도.** `useEffect` + `setState`로 정렬된 데이터를 관리하는 패턴은 의존성 배열이 복잡해지고, 디버깅이 어렵다. 순수 함수형 파이프라인(`useMemo` 체인)으로 전환하면 데이터 흐름이 더 예측 가능해질 것이다.

---

> 사용자에게 보이는 것은 "글 목록"이다. 그 뒤에서 네 개의 소스가 서로 다른 프로토콜로 데이터를 보내고, 서로 다른 타이밍에 도착하고, 서로 다른 구조를 가지고 있다는 사실은 — 사용자가 절대 알 필요 없는 복잡성이다. 그 복잡성을 코드가 흡수하는 것. 그게 이 피처의 일이었다.
