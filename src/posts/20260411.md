# 디자인 시안을 읽는 개발자 — Figma 반복 패턴을 React 선언적 API로 바꾸기

> "같은 구조가 세 번 반복되면, 그건 패턴이다. 패턴이 보이면 이름을 붙여야 한다."

---

어드민 서비스의 복잡한 생성 폼을 만들고 있었다. Figma 시안을 보면 폼이 여러 섹션으로 나뉘어 있다 — 기본정보, 조건 설정, 기간 설정, 수량 제한, 적용 대상, 메모. 각 섹션의 내용은 전부 다르지만, 눈을 가늘게 뜨고 보면 시각적 구조가 동일하다.

**제목 + 설명 + (선택) 안내 카드 + 폼 필드들.**

이 구조가 6번 반복된다. 디자이너 입장에서는 당연한 일관성이고, 개발자 입장에서는 추상화 신호다.

---

## 문제: 시안과 코드의 구조가 안 맞는다

처음에는 각 섹션을 개별적으로 마크업했다. `div` 안에 `h2`, `p`, 폼 필드들을 직접 배치하는 방식. 동작에는 문제가 없었지만, 디자이너가 "조건 설정 섹션 위에 안내 문구 추가해주세요"라고 하면 어디를 고쳐야 하는지 코드에서 바로 찾기가 어려웠다. 시안에서는 섹션 단위로 이야기하는데, 코드에서는 그 경계가 보이지 않았다.

시안의 단위와 코드의 단위가 어긋나 있었다.

---

## 관찰: 반복되는 것은 무엇인가

시안을 다시 펼쳐놓고 정리했다.

| 요소 | 필수 | 예시 |
|---|---|---|
| 제목 | O | "기본정보", "기간 설정" |
| 설명 | O | "필요한 정보를 설정해주세요." |
| 안내 카드 | X | 조건 설정 섹션의 bullet list |
| 폼 필드들 | O | 이름 input, 유형 select, ... |
| 구분선 | 대부분 | 마지막 섹션은 없음 |

이게 6개 섹션 전부에 적용된다. 차이점은 안내 카드의 유무와 구분선의 유무 정도. 나머지는 동일 구조다.

---

## 설계: 30줄짜리 컴포넌트

```tsx
type FormSectionProps = {
    title: string;
    description: string;
    formStack: React.ReactNode[];
    header?: React.ReactNode;
    className?: string;
};
```

API를 이렇게 잡았다. 핵심 결정이 세 가지 있었다.

### 왜 `children`이 아니라 `formStack: ReactNode[]`인가

이 폼의 필드들은 조건부다. 특정 유형이 선택되어야 다음 필드가 보이고, 특정 모드일 때만 추가 옵션이 나타난다. 이걸 `children`으로 처리하면 호출하는 쪽에서 조건부 렌더링을 직접 관리해야 한다.

배열로 받으면 `.filter(Boolean)`으로 falsy 값을 자동으로 걸러낼 수 있다.

```tsx
formStack={[
    <BenefitTypeSelector label="유형" />,
    selectedType && <BenefitModeSelector label="세부 설정" />,
    deliveryMethod === 'SHIPPING' && <DeliveryForm />,
]}
```

`false`, `undefined`, `null` — 조건이 거짓이면 배열에 falsy 값이 들어가고, 컴포넌트 안에서 `filter(Boolean)`이 알아서 제거한다. 호출하는 쪽은 **"이 필드는 이 조건일 때 보인다"**만 선언하면 된다.

### 왜 `header` 슬롯이 필요한가

6개 섹션 중 3개에 안내 카드가 있다. "조건 설정", "기간 설정", "수량 제한" 섹션에는 bullet list 형태의 안내문이 폼 필드 위에 놓인다. 나머지 3개에는 없다.

이걸 boolean 플래그와 데이터로 처리할 수도 있었지만, 안내 카드의 형태가 앞으로 달라질 수 있다는 판단이 있었다. `header`를 ReactNode 슬롯으로 열어두면 어떤 형태든 받을 수 있다.

### 왜 `className`에 기본값을 넣었는가

대부분의 섹션은 하단에 `border-b` 구분선이 있다. 하지만 마지막 섹션이나 일부 선택적 섹션은 구분선이 없다. 기본값을 구분선 있는 스타일로 두고, 예외인 섹션만 `className=""`으로 오버라이드하게 했다. 다수의 케이스를 기본값으로, 소수의 예외를 명시적으로.

---

## 결과: 시안의 단위 = 코드의 단위

이제 전체 폼 컴포넌트를 보면 `FormSection`이 6개 나열되어 있다. 각 섹션이 시안의 어떤 영역인지 `title`만 보면 안다.

```tsx
<FormSection
    title="조건 설정"
    description="사용할 수 있는 조건을 설정해주세요"
    header={<BulletListCard items={USAGE_CONDITIONS} />}
    formStack={[
        mode === 'external' && <GiftOptionSelector />,
        <MinimumAmountInput />,
    ]}
/>
```

디자이너가 "조건 설정 섹션에 안내문구 하나 더 추가해주세요"라고 하면, `title="조건 설정"`을 찾아서 `header`의 `items` 배열에 문자열 하나를 추가하면 끝이다. 디자이너의 언어와 코드의 구조가 같은 단위로 움직인다.

그리고 이 컴포넌트는 모노레포의 공통 패키지에 있다. 어드민과 파트너 두 앱에서 동일하게 사용한다. 폼의 구조는 같고, 어떤 필드를 보여줄지만 다르다.

---

## 추상화의 크기에 대하여

이 컴포넌트는 30줄이다. 대단한 로직이 없다. `div` 몇 개와 `map` 하나.

그런데 이 30줄이 만드는 차이는 크다. 코드의 추상화 경계가 디자인의 시각적 경계와 일치하면, **시안 변경이 코드 변경으로 번역되는 비용이 거의 0에 가까워진다.** 디자이너가 말하는 단위와 개발자가 수정하는 단위가 같으니까.

추상화의 가치는 코드의 줄 수가 아니라, **"어디를 잘라야 하는가"라는 판단**에 있다. 폼 필드 단위로 자르면 너무 작고, 폼 전체를 하나로 묶으면 너무 크다. 섹션 단위가 정확히 시안의 반복 단위와 일치했고, 그 경계에서 자른 것이 이 컴포넌트의 본질이다.

---

> 좋은 추상화는 코드를 줄이는 게 아니라, 대화를 줄인다. 디자이너가 "저 섹션"이라고 가리킬 때, 개발자도 "저 컴포넌트"를 가리킬 수 있으면 된다.
