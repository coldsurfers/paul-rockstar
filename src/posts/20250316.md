# React Native New Architecture — 브릿징의 진화

> 우리가 편하게 쓰던 `<View />`는 사실 네이티브 브릿징 컴포넌트다.

---

## 브릿징이란

React Native는 JavaScript와 네이티브 코드 간의 통신이 필요하다. 기존 방식은 **JavaScript → Bridge → 네이티브 모듈** 흐름으로 JSON 직렬화와 비동기 메시지 큐를 사용했다. 유연하지만 JS와 네이티브 간 병목이 생기는 구조였다.

`<View />`, `<TouchableOpacity />`, `AppState` — `import {} from 'react-native'`로 쓰는 대부분이 이 브릿징 위에 있다.

---

## New Architecture: JSI 기반

새 아키텍처는 **JSI(JavaScript Interface)** 를 통해 동기화 및 직접 호출이 가능해졌다.

| | Old Arch | New Arch |
|---|---|---|
| 통신 방식 | 비동기 메시지 큐 | JSI 직접 호출 |
| 직렬화 | JSON | 없음 |
| 성능 | 병목 있음 | 개선됨 |

핵심 세 가지:

- **JSI** — Bridge 대신 JS와 네이티브가 직접 소통하는 인터페이스
- **Fabric** — JSI를 통해 네이티브와 직접 소통하는 새 UI 렌더링 엔진
- **TurboModules** — 필요할 때만 로드(Lazy Loading)되는 기존 Native Modules의 개선 버전

---

## Codegen이 필요한 이유

기존에는 JS ↔ 네이티브 브릿징을 수동으로 구현해야 했다. 타입 불일치, 반복 작업, 비효율적인 연동 코드가 문제였다.

**Codegen**은 TypeScript로 선언된 인터페이스를 기반으로 C++ 바인딩 코드를 자동 생성한다. JS와 네이티브 간 타입 안전한 연결을 보장한다.

생성되는 인터페이스는 두 가지다.

**TurboModules (Native Modules)**

```tsx
export interface MyModule extends TurboModule {
  getDeviceName(): Promise<string>
}
```

Codegen이 이를 바탕으로 C++ 네이티브 코드를 생성하고, iOS/Android 구현 인터페이스를 제공한다.

**Fabric (Native UI Components)**

```tsx
export interface MyButtonProps {
  title: string
  onPress: () => void
}

export const MyButton = codegenNativeComponent<MyButtonProps>('MyButton')
```

Codegen이 C++ 바인딩을 생성하고, iOS/Android에서 네이티브 UI 컴포넌트를 직접 구현할 수 있게 한다.

## Android: JNI와 C++

Android는 Java/Kotlin이 기본 언어라 C++과의 연결에 **JNI(Java Native Interface)** 가 필요하다.

```
// Old Arch
JavaScript → (Bridge) → Java/Kotlin

// New Arch
JavaScript → (JSI) → C++ (TurboModules) → JNI → Java/Kotlin
```

JNI 함수 예시:

```cpp
#include <jni.h>

extern "C"
JNIEXPORT jstring JNICALL
Java_com_example_MyModule_getMessage(JNIEnv *env, jobject instance) {
    return env->NewStringUTF("Hello from C++!");
}
```

Java/Kotlin에서 C++ 함수를 등록:

```java
public class MyModule extends ReactContextBaseJavaModule {
    static { System.loadLibrary("mylibrary"); }

    private native void nativeFunction();

    @ReactMethod
    public void callNativeFunction() {
        nativeFunction();
    }
}
```

## iOS: Objective-C++

iOS는 Objective-C/Swift가 기본이라 JNI가 필요 없다. `.mm` 확장자(Objective-C++)를 쓰면 C++과 Objective-C를 혼합할 수 있다.

C++ 구현:

```cpp
// MyNativeModule.cpp
std::string MyNativeModule::getMessage() {
    return "Hello from C++!";
}
```

Objective-C++에서 C++ 호출 후 React Native로 전달:

```objc
// MyNativeModule.mm
RCT_EXPORT_METHOD(getMessage:(RCTPromiseResolveBlock)resolve
                  rejecter:(RCTPromiseRejectBlock)reject) {
    std::string message = MyNativeModule::getMessage();
    NSString *nsMessage = [NSString stringWithUTF8String:message.c_str()];
    resolve(nsMessage);
}
```

---

Fabric과 TurboModules는 이제 기본 아키텍처로 자리잡았다. Old Arch는 Legacy가 되어가고 있고, 커뮤니티의 주요 네이티브 모듈들도 대부분 New Arch를 지원한다.

*No Free Lunch. 성능 개선에는 항상 복잡도가 따라온다.*
