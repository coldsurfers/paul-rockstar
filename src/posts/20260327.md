## 1개월차: 낯선 코드베이스를 내 손에 맞게

입사하고 첫 주에 받은 코드는 Expo Router 기반 React Native 앱이었다. 이미 어느 정도 돌아가고 있었지만, 군데군데 구조적으로 불편한 지점들이 보였다.

처음 건드린 건 **SafeArea** 였다.

루트 레이아웃에 `SafeAreaView`가 박혀 있었는데, 스크린마다 safe area가 다르게 필요한 상황이 생기면 전역으로 예외처리를 해야 했다. 이름부터 이미 신호를 주고 있었다.

```ts
// 이런 이름이 생기기 시작하면 구조를 바꿔야 한다는 신호다
GLOBAL_BOTTOM_SAFE_AREA_EXCLUDE_PATH_LIST
```

쳐냈다. 각 스크린이 자기 safe area를 책임지는 방식으로 이관했다. NavBar도 `navigation.setOptions` 패턴으로 바꾸면서 깜빡임이 사라졌다. 전역에서 조율하려 했던 것들이 각자 위치에서 처리되니 훨씬 명확해졌다.

**ProtectedLink**는 좀 재미있는 작업이었다. 인증이 필요한 화면으로 이동할 때 조건부로 리다이렉트하는 컴포넌트인데, hook 방식과 컴포넌트 방식을 함께 제공했다.

```tsx
// before — 사용하는 쪽이 인증 로직을 알아야 한다
<Link href="/mypage" onPress={checkAuth}>마이페이지</Link>

// after — 의도만 선언한다
<ProtectedLink href="/mypage">마이페이지</ProtectedLink>
```

코드가 줄었다는 게 중요한 게 아니라, **의도가 명확해졌다**는 게 중요하다.

**Type-safe search params**도 이 시기에 정리했다. `expo:typegen`으로 라우트 타입을 자동 생성하고, `RouteSearchParams`를 재사용 구조로 만들었다. 타입 없이 `string`으로 받던 파라미터들이 컴파일 단에서 잡히기 시작했다.

## 개발 환경도 제품이다

코드만 만지지 않았다. **개발 환경** 자체를 제품처럼 다뤘다.

- `lint-staged` → `lefthook` 으로 교체. 더 빠르고, config가 명확하다.
- Lint CI GitHub Actions 추가. PR 올리면 자동으로 린트가 돈다.
- AI 기반 코드 리뷰 자동화. PR 코멘트 이벤트에 트리거를 걸었다.
- 아이콘 생성 스크립트 마이그레이션. 기존 방식의 의존성 문제를 걷어냈다.
- Sentry release 속성 추가. 배포된 버전이 에러 로그에 찍힌다.

이런 작업들은 PR 숫자에 잘 안 잡힌다. 근데 팀 전체 속도에는 확실히 영향을 준다. **자동화가 한 번 깔리면 이후엔 신경을 안 써도 된다.** 그게 핵심이다.

## 2개월차: 쿠폰 시스템을 처음부터 끝까지

2월 말부터 어드민 작업을 시작했다. 주제는 **쿠폰**.

처음에는 단순해 보였다. 쿠폰 생성 폼 하나 만들면 되는 거 아닌가. 그런데 파고들수록 레이어가 쌓였다.

- 마케팅용 vs 복지용 → 같은 폼, 다른 필드
- 운영자 앱 vs 파트너 앱 vs 가맹점 앱 → 같은 도메인, 다른 컨텍스트
- 생성 → 발급 내역 → 사용 내역 → 상태 관리 → 적용 대상

4일에 커밋 50건. 이후 3주 더 이어졌다.

이 작업에서 가장 중요한 설계 결정은 **도메인 응집**이었다.

```
packages/common/components/src/domains/coupon/
├── coupon-template-form/              ← 마케팅/복지 통합 폼
├── coupon-template-form-provider/     ← RHF Context
├── coupon-sticky-preview-card/        ← 실시간 프리뷰
├── coupon-template-detail/            ← 상세 뷰
└── coupon-apply-target/               ← Funnel 기반 적용 대상
```

세 앱이 `@common/components/domains/coupon`을 가져다 쓴다. 앱마다 쿠폰 폼을 따로 만들지 않는다. 한 번 정의하고 세 군데서 쓴다.

**CouponFormSection** 패턴이 특히 깔끔했다. 각 섹션(유효기간, 발급 제한, 사용 조건)이 독립적인 단위로 존재한다. 교체도 되고 테스트도 쉽다.

**Funnel 기반 적용 대상 Drawer**는 다단계 선택 흐름을 Funnel 패턴으로 관리했다. 대상 선택 → 필터 → 세부 항목 선택. 상태가 `FunnelContext` 안에서 흐른다. Drawer 자체도 공통화해서 상세 뷰에서도 그대로 재사용된다.

실시간 프리뷰 카드는 `FormContext`를 같이 구독한다. 폼 입력이 즉시 카드에 반영된다. 작은 디테일인데, 쿠폰을 만드는 사람이 결과물을 바로 보면서 입력할 수 있다는 게 중요하다. **만드는 사람의 경험이 쓰는 사람의 신뢰로 이어진다.**

## 3일 만에 AI 파이프라인

3월 말, 3일간 내부 개발자 생산성을 위한 AI 도구를 만들었다.

- CLI + 웹 채팅 UI
- MCP 서버 연동
- 로컬 임베딩 기반 RAG 파이프라인 (벡터 DB 연동)
- 클라우드 배포까지

내용보다 **방식**을 얘기하고 싶다.

기획서가 없었다. 필요하다고 판단했고, 가정을 세웠고, 만들었다. 3일이라는 시간 안에 배포까지 나온 건 기술 스택이 익숙해서가 아니라, **범위를 명확히 쳐냈기 때문**이다. 지금 당장 필요한 것만 만든다. 나머지는 나중에.

## 2개월을 돌아보며

기술적으로 정리하면:

| 영역 | 키워드 |
|------|--------|
| 앱 구조 | SafeArea 이관, NavBar 패턴, ProtectedLink, Type-safe params |
| 개발 환경 | lefthook, Lint CI, AI 리뷰 자동화, Sentry |
| 어드민 | 쿠폰 도메인 레이어, Funnel Drawer, 실시간 프리뷰 |
| AI 도구 | CLI, MCP, RAG 파이프라인 |

그런데 숫자보다 더 중요한 건 **각 결정이 제품 방향과 얼마나 맞았냐**다.

SafeArea를 스크린으로 이관한 건 단순히 코드 정리가 아니었다. 스크린마다 다른 레이아웃이 필요하다는 제품 요구에 맞는 구조를 만든 거다. 쿠폰 도메인 레이어를 공통으로 뺀 건 앱이 세 개라는 현실을 반영한 설계 결정이었다.

코드는 제품을 만들기 위한 브릿지 작업이다. 브릿지가 튼튼해야 건너편에 닿는다. 그런데 브릿지 자체가 목적이 되면 안 된다.

2개월차에 이걸 다시 한번 확인했다.
