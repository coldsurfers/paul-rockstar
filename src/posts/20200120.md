# 이벤트 루프 — 자바스크립트가 비동기를 처리하는 방식

> 자바스크립트는 싱글 스레드다. 그런데 어떻게 비동기가 가능한가.

---

## 자바스크립트의 세 가지 구성 요소

- **Call Stack** — 현재 실행 중인 함수들의 스택
- **Task Queue** — 비동기 콜백들이 대기하는 큐
- **Heap Memory** — 객체가 할당되는 구조화되지 않은 메모리 공간

Call Stack이 비어야 Task Queue의 항목이 실행된다. 이게 전부다.

---

## 이벤트 루프의 역할

이벤트 루프는 Call Stack을 감시한다. Stack이 비어 있으면 Task Queue에서 꺼내 Stack에 올린다.

```js
console.log('Hi!');

setTimeout(() => {
  console.log('Yay!');
}, 4000);

console.log('Bye!');

// 출력: Hi! → Bye! → (4초 후) Yay!
```

`setTimeout`은 Web API다. 타이머가 끝나면 콜백을 Task Queue에 쌓는다. Stack이 비면 이벤트 루프가 꺼내 실행한다.

`fetch`도 동일하다.

```js
console.log('Hi! again');

fetch(url).then(() => console.log('Fetched!'));

console.log('Bye! again');

// 출력: Hi! again → Bye! again → Fetched!
```

XHR이 완료되면 `.then` 콜백이 Task Queue에 들어가고, Stack이 비면 실행된다.

---

## setTimeout(fn, 0)이 즉시 실행되지 않는 이유

```js
console.log('Oops!');
setTimeout(() => console.log('I waited 0ms!'), 0);
console.log('Last');

// 출력: Oops! → Last → I waited 0ms!
```

0ms는 "0ms 후에 즉시 실행"이 아니라 "비동기로 처리"를 의미한다. Task Queue를 거치기 때문에 동기 코드가 모두 끝난 뒤에야 실행된다.

---

## 렌더링도 비동기다

브라우저 렌더링은 Web API의 일종이다. 매 16ms(약 60fps)마다 Render Queue에 쌓인다.

Render Queue는 Task Queue보다 우선순위가 높다. 하지만 Call Stack의 동기 코드보다는 낮다.

Call Stack에 무거운 동기 함수가 쌓이면 렌더링이 지연된다. 화면이 버벅이는 이유다. 오래 걸리는 계산은 비동기로 처리해야 렌더링을 막지 않는다.

---

## Web API는 누가 실행하나

`setTimeout`, `fetch`, 이벤트 리스너 — 이것들은 모두 JavaScript 엔진이 아닌 브라우저 API가 처리한다. Chrome은 V8 엔진 위에서 이 API들을 별도로 실행하고, 완료되면 콜백을 Task Queue에 넣는다.

JavaScript는 싱글 스레드지만 브라우저는 아니다.
