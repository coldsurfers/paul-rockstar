# Virtual DOM — 왜 필요한가

> 브라우저 DOM을 직접 건드리는 것은 생각보다 훨씬 비싸다.

---

## DOM이란

DOM(Document Object Model)은 HTML 문서를 트리 구조로 표현한 API다. JavaScript가 HTML을 읽고 조작할 수 있게 해주는 다리 역할을 한다.

브라우저는 HTML을 렌더링할 때 이 순서를 따른다.

```
HTML 파싱 → DOM Tree 구성 → 렌더 트리 생성 → 레이아웃 → 화면에 그리기
```

문제는 **노드가 변경될 때마다 이 과정이 반복된다**는 것이다. 변경이 잦을수록 브라우저가 받는 부하는 계속 쌓인다.

---

## Virtual DOM의 역할

Virtual DOM은 트리 구조를 가진 순수 JavaScript 객체다. 메모리 안에만 존재하고, 실제 화면에는 그려지지 않는다.

새 빌딩을 짓는 것과 청사진을 그리는 것의 차이다.

변화가 생기면 먼저 Virtual DOM에 적용하고, 실제 DOM에는 최소한의 변경만 한 번에 반영한다. 이것이 핵심이다.

---

## React의 Reconciliation

state가 바뀌면 React는 새로운 Virtual DOM Tree를 생성한다. 그리고 이전 Tree와 비교(diff)해서 달라진 부분만 실제 DOM에 패치한다.

일반적인 두 트리 비교 알고리즘은 `O(n³)` 복잡도를 가진다. React는 두 가지 가정을 통해 이를 `O(n)`으로 줄인다.

**가정 1** — 타입이 다른 엘리먼트는 다른 트리를 만든다. Root 타입이 다르면 서브트리 전체를 교체한다.

**가정 2** — `key`가 자식 엘리먼트 변화의 힌트가 된다. 안정적이고 예측 가능한 `key`가 있어야 어떤 노드만 추가/제거됐는지 알 수 있다.

```jsx
// key 없이 새 항목 추가 → 전체 재렌더
<ul>
  <li>John</li>
  <li>Alice</li>
</ul>

// key 있음 → Ed만 추가
<ul>
  <li key="9">John</li>
  <li key="10">Alice</li>
  <li key="11">Ed</li>
</ul>
```

`Math.random()` 같은 예측 불가능한 key는 오히려 성능을 망가뜨린다.

---

## 그 다음 — React Fiber

React Fiber는 Reconciliation 알고리즘을 재작성한 것이다. DOM을 순환하는 대신, fiber라는 새 데이터 구조를 사용한다. Fiber는 parent-child 관계를 linked-list로 트래킹하며, 작업을 쪼개서 스케줄링하는 "incremental rendering"을 가능하게 한다.

---

*Virtual DOM은 마법이 아니다. 비싼 DOM 조작을 줄이기 위한 영리한 배치 전략이다.*
