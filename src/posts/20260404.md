# RAG가 "쓰이기 시작한" 다음에 오는 일

> 봇이 답하기 시작하면, 다음 질문은 "더 잘 답하게 하려면 뭘 넣어야 하지?"가 된다. 그리고 거기서부터가 진짜 운영이다.

---

## 이전 글에서 남긴 질문

[지난번](/blog/20260331) 슬랙 RAG 봇을 팀에 올리고 나서, 마지막에 이런 말을 했다.

> "AI를 써볼까?"가 아니라, "이제 이걸 어디까지 연결할까?"로 질문이 바뀐다.

실제로 바뀌었다. 그리고 그 다음에 온 건 모델을 바꾸는 일도, 프롬프트를 고치는 일도 아니었다.

**문서를 어떻게 넣을 것인가.**

봇이 답하려면 근거가 있어야 하고, 근거는 결국 인제스천 파이프라인의 문제다. 데모에서는 파일 몇 개 밀어 넣으면 되지만, 팀이 매일 쓰는 도구가 되면 이야기가 달라진다.

---

## "문서 넣기"가 왜 복잡해지는가

처음에는 단순했다. `context/` 폴더에 마크다운을 넣고, 임베딩 돌리고, 벡터 DB에 저장한다. 끝.

그런데 문서가 늘어나면 바로 문제가 생긴다.

- 같은 문서를 두 번 넣으면 검색 결과가 중복된다
- 폴더 구조가 `context/`, `web/docs/`로 나뉘어 있으면 어디가 최신인지 모른다
- 한 번 넣은 문서를 갱신하고 싶을 때 기존 것을 어떻게 지우는가
- 임베딩 API를 한꺼번에 쏘면 레이트 리밋에 걸린다

이런 건 RAG 아키텍처 글에서는 잘 안 다룬다. 그런데 실제로 팀에 붙이면 첫 주 안에 다 만난다.

---

## 인제스천을 "다시 돌릴 수 있는 작업"으로 만들기

가장 먼저 한 건 문서 디렉토리를 하나로 합친 것이다.

`context/`와 `web/docs/`에 흩어져 있던 문서를 `docs/` 하나로 통합했다. 단순한 리팩토링처럼 보이지만, 이게 안 되면 "지금 봇이 뭘 알고 있는가"를 아무도 답할 수 없다.

그 다음은 CLI를 만들었다.

```bash
# 문서를 임베딩해서 DB에 넣는다
node cli.js ingest

# 테이블을 비우고 처음부터 다시 넣는다
node cli.js ingest --clean
```

`--clean`이 왜 필요하냐면, 문서를 수정하거나 삭제한 뒤 "지금 DB 상태가 소스와 일치하는가?"를 확신할 수 없을 때가 오기 때문이다. 멱등하게 밀어 넣을 수 있어야 운영이 된다.

문서 삽입도 단순 insert가 아니라 **upsert**로 바꿨다. 같은 문서를 다시 넣으면 갱신되지, 중복 행이 생기지 않는다. 사소해 보이지만 이게 없으면 검색 결과에 같은 문단이 두세 번 뜨는 걸 보게 된다.

---

## 임베딩 API는 무한하지 않다

문서가 수십 개일 때는 상관없지만, 수백 개가 되면 임베딩 API 호출이 몰린다.

Gemini 임베딩 API 기준 100 RPM. 생각보다 빨리 닿는다. 한꺼번에 쏘면 429가 오고, 재시도 로직이 없으면 파이프라인이 중간에 멈춘다.

그래서 임베딩 모듈을 **레이트 리밋 임베더**로 교체했다. 호출 간격을 RPM 기준으로 조절하고, 한 번의 ingest 명령으로 끝까지 돌아가게 만들었다.

모델 성능이나 검색 알고리즘보다, 이런 **호출 한도가 실제 운영의 병목**인 경우가 많다. 데모에서는 절대 안 보이는 부분이다.

---

## "팀 문서" 다음에 오는 코퍼스: 법령

봇이 팀 내부 문서를 잘 찾아주기 시작하면, 자연스럽게 다음 질문이 온다.

**"이건 법적으로 어떻게 되는 거야?"**

우리 서비스는 의료·미용 도메인이다. 의료기기법, 개인정보보호법, 의료 데이터 관련 규제가 실무에 직접 붙어 있다. 팀원이 슬랙에서 "이 데이터 수집이 개인정보보호법 몇 조에 해당하지?"라고 물을 수 있다면, 그건 꽤 쓸모 있다.

물론 법률 자문을 대체할 수는 없다. 하지만 **"일단 관련 조문을 찾아서 읽어보는 시간"** 을 줄여줄 수는 있다.

그래서 한국 법령 텍스트를 RAG 파이프라인에 올리는 작업을 시작했다.

---

## 법령 텍스트는 팀 문서와 다르다

법령을 넣으려고 보니, 팀 마크다운과는 성격이 완전히 다르다.

- 조·항·호의 계층 구조가 있다
- 하나의 법률이 수십~수백 조로 이루어진다
- 개정 이력이 있고, "현행"이 뭔지가 중요하다
- 원문 그대로 넣으면 LLM이 읽기 어려운 형태다

그래서 접근을 나눴다.

1. **법령 fetch 커맨드와 매니페스트**: 어떤 법령을 가져올지 목록으로 관리하고, CLI로 원문을 내려받는다
2. **LLM 친화적 변환**: 원문의 조·항·호 구조를 유지하면서, 모델이 인용할 수 있는 형태로 정리한다
3. **Cloudflare 기반 인제스천**: 법령은 팀 문서보다 양이 많으므로, 인프라도 맞춰 준비한다

아직 완성은 아니다. 하지만 "어떤 법령을, 어떤 형태로, 어디에 넣을지"의 뼈대는 잡혔다.

---

## 넣지 않는 것도 설계다

전편에서도 썼지만, 이번에도 같은 원칙이 반복됐다.

의료 데이터 컴플라이언스 관련 정리를 하면서 느낀 건, 법령 코퍼스에도 **경계**가 필요하다는 것이다.

- 모든 법률을 다 넣을 필요는 없다. 서비스와 직접 관련 있는 것만 넣는다
- 법령을 근거로 답하되, "법적 판단"처럼 읽히면 안 된다
- 민감 데이터 처리 조항은 참조용으로 넣되, 고객 데이터 자체는 여전히 제외한다

RAG의 범위를 넓히는 작업은 결국 **"여기까지는 답하고, 여기서부터는 답하지 않는다"** 의 선을 계속 다시 긋는 일이다.

---

## 지금 파이프라인의 모습

정리하면 이렇다.

```
docs/                ← 단일 문서 소스
  ├── internal/      ← 팀 내부 문서
  └── legal/         ← 법령·규제 문서

CLI (ingest)
  ├── --clean        ← 테이블 리셋 후 재삽입
  ├── upsert         ← 중복 방지
  └── rate limiter   ← RPM 기준 임베딩 호출 조절

Vector DB
  └── Slack bot, MCP 등에서 검색
```

화려한 아키텍처는 아니다. 하지만 **다시 돌릴 수 있고, 깨지지 않고, 무엇이 들어 있는지 알 수 있다.** 운영 가능한 파이프라인이 갖춰야 할 최소한은 이 정도라고 생각한다.

---

## 결국

RAG는 모델이 똑똑해지는 게 아니라 **근거가 쌓이는 속도와 품질**로 결정된다.

모델은 바꾸면 되지만, 파이프라인이 깨지면 봇은 아무것도 모르는 상태로 돌아간다. 그래서 이번 주에 한 일의 대부분은 모델 쪽이 아니라 **문서가 들어오고 정리되는 경로**를 다듬는 일이었다.

전편에서 "쓰이기 시작했다"고 했는데, 쓰이기 시작한 도구가 계속 쓰이려면 결국 이런 작업이 따라온다. 눈에 안 보이고, 데모에도 안 나오고, 기술 블로그에서도 잘 안 다루는 부분.

하지만 이게 없으면 봇은 멈추고, 있으면 봇은 조금씩 더 많이 안다.

그 차이를 만드는 건 프롬프트 엔지니어링이 아니라, 매일 돌아가는 파이프라인이다.
