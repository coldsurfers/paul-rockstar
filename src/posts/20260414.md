# CLI 위에 SDK를 얹다 — XML 응답을 TypeScript 표준으로 바꾸는 과정

> "도구를 만들었으면, 다음은 그 도구 위에 올라갈 코드를 생각할 차례다."

---

[지난 글](/posts/20260410)에서 KOPIS 공공 API를 CLI로 감쌌다. 터미널에서 바로 공연 데이터를 조회할 수 있게 된 건 좋았지만, 하나가 더 남아 있었다. COLDSURF 서비스 코드에서 이 데이터를 프로그래밍 방식으로 가져다 쓰려면 — 결국 또 XML을 직접 파싱해야 한다.

CLI는 사람을 위한 인터페이스다. SDK는 코드를 위한 인터페이스다. CLI만 만들고 끝내면, 다른 프로젝트에서 KOPIS 데이터가 필요할 때마다 XML 파싱 코드를 다시 작성하게 된다.

---

## 문제: XML이 만드는 DX 지옥

KOPIS API의 응답은 이렇게 생겼다.

```xml
<dbs>
  <db>
    <mt20id>PF254783</mt20id>
    <prfnm>2026 서울재즈페스티벌</prfnm>
    <prfpdfrom>20260515</prfpdfrom>
    <prfpdto>20260517</prfpdto>
    <fcltynm>올림픽공원</fcltynm>
    <genrenm>대중음악</genrenm>
    <prfstate>공연예정</prfstate>
    <openrun>N</openrun>
  </db>
</dbs>
```

이걸 코드에서 쓰려면 매번 같은 의식을 치른다.

`fetch`로 XML 문자열을 받는다. 파서를 돌린다. `parsed.dbs.db`에 접근하는데, 결과가 하나면 객체고 여러 개면 배열이다. 매번 `Array.isArray` 체크가 따라붙는다. `mt20id`가 공연 ID라는 걸 문서를 보고 확인한다. `prfnm`이 공연명이라는 걸 문서를 보고 확인한다. `prfstate`가 `'01'`이면 공연예정이라는 것도 문서를 보고 확인한다.

**타입 안전성은 없다. 자동완성도 없다. 실수를 컴파일 타임에 잡을 방법도 없다.**

한두 번이면 참을 수 있다. 하지만 공연 데이터를 다루는 곳은 계속 늘어난다. 서버 사이드 파이프라인. 어드민 대시보드. 추천 시스템 실험. 매번 같은 XML 파싱 코드를 복붙하는 건 시간 낭비이자 버그의 온상이다.

## 설계 판단: 패키지를 분리하지 않는다

SDK를 만들기로 결정하고 첫 번째로 마주친 질문은 이거였다. **기존 `@coldsurf/tickets` CLI 패키지에 SDK를 같이 넣을 것인가, 별도 패키지로 분리할 것인가.**

분리하면 깔끔하다. SDK 사용자가 `commander`나 `cli-table3` 같은 CLI 의존성을 설치할 필요가 없다.

하지만 현재 상황은 다르다. SDK의 핵심 의존성은 `fast-xml-parser` 하나뿐이다. CLI 의존성은 side-effect가 없어서, 설치되어 있어도 import 하지 않으면 문제없다. 반면 패키지를 분리하면 모노레포에 패키지가 하나 늘고, turbo 빌드 의존성 설정이 필요하고, 버전 관리가 이중으로 생긴다.

깔끔함을 위해 복잡함을 추가하는 건 본말이 전도된 거다. **변경 범위를 최소화하는 게 이겼다.** 하나의 패키지에서 dual entry point로 CLI와 SDK를 모두 제공하기로 했다.

```typescript
// tsup.config.ts
export default defineConfig([
  // SDK — 타입 선언 포함
  {
    entry: ['src/index.ts'],
    format: ['esm'],
    dts: true,
    clean: true,
  },
  // CLI — shebang 포함
  {
    entry: ['src/cli.ts'],
    format: ['esm'],
    banner: { js: '#!/usr/bin/env node' },
  },
]);
```

`package.json`의 `main`과 `types`는 SDK entry를 가리킨다. `bin`은 CLI entry를 가리킨다. `import`하면 SDK가 오고, `npx`로 실행하면 CLI가 온다. 같은 패키지, 두 개의 문.

## 핵심: Raw 인터페이스를 감추고, 의미 있는 타입을 노출한다

SDK의 본질은 변환 레이어다. KOPIS API가 돌려주는 암호 같은 필드명을 사람이 읽을 수 있는 이름으로 바꾼다.

```typescript
// 내부에서만 쓰이는 Raw 타입 — SDK 소비자는 볼 일이 없다
interface RawListItem {
  mt20id: string;
  prfnm: string;
  prfpdfrom: string;
  fcltynm: string;
  genrenm: string;
  prfstate: string;
}

// SDK가 노출하는 타입 — 이름만 보면 뭔지 안다
export interface KopisPerformance {
  id: string;
  title: string;
  startDate: string;
  endDate: string;
  venue: string;
  genre: string;
  state: string;
}
```

변환 함수는 단순하지만 중요한 일을 한다.

```typescript
function toPerformance(raw: RawListItem): KopisPerformance {
  return {
    id: String(raw.mt20id),
    title: String(raw.prfnm),
    startDate: String(raw.prfpdfrom),
    endDate: String(raw.prfpdto),
    venue: String(raw.fcltynm),
    genre: String(raw.genrenm),
    state: String(raw.prfstate),
  };
}
```

`String()` 래핑은 방어적이다. XML 파서가 숫자처럼 생긴 문자열을 `number`로 변환할 수 있기 때문이다. 공연 ID `PF254783`은 괜찮지만, 날짜 `20260515`는 파서에 따라 숫자가 될 수 있다. 모든 필드를 명시적으로 `String()`하면 타입이 항상 보장된다.

## 배열/단일 객체 문제의 표준화

XML 파싱에서 가장 흔하게 버그를 만드는 패턴이 있다. **결과가 하나일 때는 객체, 여러 개일 때는 배열로 파싱되는 것.**

```typescript
// 결과 1건: parsed.dbs.db = { mt20id: '...', prfnm: '...' }
// 결과 2건: parsed.dbs.db = [{ mt20id: '...' }, { mt20id: '...' }]
```

이걸 매번 호출하는 쪽에서 처리하게 두면, 실수는 시간문제다. SDK가 이 분기를 한 번에 해결한다.

```typescript
const db = parsed?.dbs?.db;
if (!db) return [];
const items: RawListItem[] = Array.isArray(db) ? db : [db];
return items.map(toPerformance);
```

**SDK 소비자는 항상 배열을 받는다.** 결과가 0건이면 빈 배열, 1건이면 요소 하나짜리 배열, N건이면 N개 배열. 분기 로직을 생각할 필요가 없다.

같은 패턴이 중첩 데이터에서도 반복된다. 공연 상세의 티켓 판매처(`relates`), 상세 이미지(`styurls`), 공연장의 홀 목록(`mt13s`) — 전부 단일/배열 분기가 필요한 곳이다. SDK가 이걸 모두 표준화한다.

```typescript
function parseTickets(
  relates?: { relate: RawRelate | RawRelate[] }
): KopisTicketInfo[] {
  if (!relates?.relate) return [];
  const list = Array.isArray(relates.relate)
    ? relates.relate
    : [relates.relate];
  return list
    .filter((r) => r.relatenm && r.relateurl)
    .map((r) => ({ seller: r.relatenm, url: r.relateurl }));
}
```

## 상수를 코드로 만들면 문서가 필요 없어진다

`CCCD`가 대중음악이고, `AAAA`가 연극이고, `GGGA`가 뮤지컬이라는 걸 외우는 사람이 있을까? KOPIS API의 카테고리 코드는 그 자체로 암호다. 매번 문서를 찾아봐야 한다.

SDK는 이 코드들을 `as const` 객체로 내보낸다.

```typescript
export const KOPIS_CATEGORIES = {
  대중음악: 'CCCD',
  연극: 'AAAA',
  '서양음악(클래식)': 'CCCA',
  뮤지컬: 'GGGA',
} as const;

export type KopisCategoryCode =
  (typeof KOPIS_CATEGORIES)[keyof typeof KOPIS_CATEGORIES];
```

IDE에서 `KOPIS_CATEGORIES.`을 치면 카테고리 목록이 자동완성된다. 문서를 열 필요가 없다. 잘못된 코드를 넣으면 타입 에러가 난다. **문서를 읽는 대신, 컴파일러가 알려준다.**

지역 코드(`KOPIS_AREAS`), 공연 상태(`KOPIS_PERFORM_STATES`), 공연장 유형(`KOPIS_VENUE_TYPES`), 시군구 코드(`KOPIS_SUB_AREAS`)까지 — KOPIS API에서 쓰이는 모든 열거형 값이 상수로 정의되어 있다.

## 최종 형태: Before / After

**Before — SDK 없이 KOPIS API를 직접 호출:**

```typescript
import { XMLParser } from 'fast-xml-parser';

const res = await fetch(
  `http://www.kopis.or.kr/openApi/restful/pblprfr?service=${key}&stdate=20260101&eddate=20261231&shcate=CCCD&signgucode=11&rows=20`
);
const xml = await res.text();
const parsed = new XMLParser().parse(xml);
const db = parsed?.dbs?.db;
const items = Array.isArray(db) ? db : db ? [db] : [];
const performances = items.map((item) => ({
  id: item.mt20id,
  title: item.prfnm,
  // ... 나머지 필드 매핑을 문서 보면서 수동으로
}));
```

**After — SDK 사용:**

```typescript
import {
  createKopisClient,
  KOPIS_CATEGORIES,
  KOPIS_AREAS,
} from '@coldsurf/tickets';

const client = createKopisClient(process.env.KOPIS_KEY!);

const performances = await client.getPerformanceList({
  startDate: '20260101',
  endDate: '20261231',
  category: KOPIS_CATEGORIES.대중음악,
  area: KOPIS_AREAS.서울,
  rows: 20,
});

// performances[0].title — 타입 안전, 자동완성 지원
```

**XML 파싱은 사라졌다.** 필드명 매핑도 사라졌다. `Array.isArray` 분기도 사라졌다. 남은 건 의도를 표현하는 코드뿐이다.

## `KopisClient` 인터페이스 — 표면적을 명확히 한다

CLI에서는 `createKopisClient`의 반환 타입이 추론으로 충분했다. 하지만 SDK로 노출하면 이야기가 달라진다. 소비자가 클라이언트 타입을 변수에 명시하거나, 함수 파라미터로 받아야 할 수 있다. 추론에 기대면 내부 구현이 바뀔 때마다 소비자의 타입이 깨진다.

```typescript
export interface KopisClient {
  getPerformanceList(params: ListParams): Promise<KopisPerformance[]>;
  getPerformanceDetail(id: string): Promise<KopisPerformanceDetail | null>;
  getVenueList(params: VenueListParams): Promise<KopisVenue[]>;
  getVenueDetail(id: string): Promise<KopisVenueDetail | null>;
  getPromoterList(params: PromoterListParams): Promise<KopisPromoter[]>;
  getAwardList(params: ListParams): Promise<KopisAwardPerformance[]>;
  getFestivalList(params: ListParams): Promise<KopisFestivalPerformance[]>;
  getCreatorList(params: ListParams): Promise<KopisCreatorPerformance[]>;
}
```

이 인터페이스가 SDK의 전체 표면적이다. 8개 메서드. 공연, 공연장, 기획사, 수상작, 축제, 원작자 — KOPIS API의 모든 도메인을 커버한다. 각 메서드의 시그니처만 보면 무엇을 넣고 무엇을 받는지 알 수 있다.

## 되돌아보며

CLI를 만들 때 이미 XML 파싱 로직과 타입 정의가 있었다. SDK를 붙이는 건 사실상 **기존 코드에 문 하나를 더 다는 작업**이었다. 변경한 파일은 4개뿐이다.

| 파일 | 변경 |
|------|------|
| `src/index.ts` (신규) | SDK public API re-export |
| `src/kopis/client.ts` | `KopisClient` 인터페이스 추가 |
| `tsup.config.ts` | dual entry point |
| `package.json` | `main`, `types`, `exports` |

**기존 CLI 코드는 한 줄도 바뀌지 않았다.**

왜 이렇게 적은 변경으로 끝났을까? 첫 설계에서 API 클라이언트와 CLI 커맨드를 분리해뒀기 때문이다. `client.ts`는 처음부터 CLI에 의존하지 않았다. SDK entry point는 이미 독립적이었던 모듈을 밖으로 내보내는 것만으로 완성됐다.

**관심사를 분리해두면, 새로운 인터페이스를 얹는 비용이 거의 0에 수렴한다.**

---

> 좋은 SDK는 문서를 읽지 않아도 쓸 수 있는 SDK다. 타입이 문서고, 상수가 가이드고, 컴파일러가 리뷰어다.
