# 쿠키를 제대로 쓰고 있는가 — SameSite부터 RBAC까지

> "쿠키를 어떻게 쓰고 계신가요?" 이 질문에 당당하게 설명할 수 있는 사람은 생각보다 많지 않다.

---

`httpOnly`, `secure`, `sameSite` 옵션을 설정하면서도 왜 그래야 하는지 깊이 고민해본 적이 없다면 — 이 글이 그 답이다.

---

## SameSite가 필요한 이유: CSRF

A.com에서 로그인하면 `access_token` 쿠키가 생성된다. 이때 악의적인 B.com이 아래 요청을 사용자 몰래 실행하면:

```html
<form action="https://A.com/api/send-money" method="POST">
```

브라우저는 A.com 쿠키를 자동으로 포함해 요청을 보낸다. 사용자 의지와 상관없이 민감한 동작이 실행된다. 이것이 **CSRF(Cross Site Request Forgery)** 다.

`SameSite` 정책은 브라우저가 쿠키를 언제 전송할지 제어한다.

| 값 | 동작 |
|---|---|
| `Strict` | 자신의 사이트 요청에만 전송 |
| `Lax` | 기본값. GET 기반 요청만 허용 |
| `None` | 모든 요청 허용. `Secure: true` 필수 |

---

## SameSite=None은 위험한가

단독으로는 그렇다. 하지만 요즘은 **Authorization 헤더 기반 JWT 인증**을 병행한다.

```
Authorization: Bearer <token>
```

브라우저는 Authorization 헤더를 스스로 생성하지 못한다. 사용자가 직접 fetch 요청을 만들어야만 인증을 통과할 수 있다. CSRF에 강한 이유다.

결론:
- **쿠키** — SSR 로그인 상태 유지
- **Authorization 헤더** — 실제 API 인증
- **SameSite=None + Secure + httpOnly** — 이 조합으로 보안성과 유연성을 동시에 확보

## SSR에서 쿠키로 로그인 상태 유지

Next.js App Router에서 서버 컴포넌트는 쿠키를 직접 읽을 수 있다.

```tsx
// app/layout.tsx
import { cookies } from 'next/headers'
import { decodeJwt } from '@/libs/utils/jwt'

export default async function Layout({ children }: { children: React.ReactNode }) {
  const token = cookies().get('access_token')?.value
  let user = null

  if (token) {
    try {
      user = decodeJwt(token)
    } catch {
      // 만료 또는 위조
    }
  }

  return (
    <html>
      <body>
        {user ? <Header user={user} /> : <Header />}
        {children}
      </body>
    </html>
  )
}
```

외부 API 호출보다 JWT 디코딩이 빠르고 효율적이다.

## 로그인 / 로그아웃 구현

**로그인** — `Set-Cookie`로 accessToken 저장:

```tsx
// app/api/auth/signin/route.ts
const cookie = serialize('access_token', authToken.accessToken, {
  httpOnly: true,
  secure: true,
  sameSite: 'none',
  path: '/',
  maxAge: 60 * 60 * 24 * 7,
  domain: process.env.NODE_ENV === 'development' ? undefined : '.yourdomain.com',
})

return new Response(null, { status: 200, headers: { 'Set-Cookie': cookie } })
```

**로그아웃** — `maxAge: 0`으로 즉시 만료:

```tsx
const expiredCookie = serialize('access_token', '', {
  httpOnly: true,
  secure: true,
  sameSite: 'none',
  path: '/',
  maxAge: 0,
  domain: process.env.NODE_ENV === 'development' ? undefined : '.yourdomain.com',
})
```

**401 자동 로그아웃**:

```tsx
// libs/fetchWithAuth.ts
export async function fetchWithAuth(input: RequestInfo, init: RequestInit = {}) {
  const res = await fetch(input, { ...init, credentials: 'include' })

  if (res.status === 401) {
    await fetch('/api/auth/signout', { method: 'POST' })
    window.location.href = '/login?expired=1'
    return null
  }

  return res.json()
}
```

## Middleware로 라우트 보호

```tsx
// middleware.ts
import { NextRequest, NextResponse } from 'next/server'
import { jwtVerify } from 'jose'

const COOKIE_ACCESS_TOKEN_KEY = 'access_token'

export async function middleware(req: NextRequest) {
  const token = req.cookies.get(COOKIE_ACCESS_TOKEN_KEY)?.value

  if (!token) {
    return NextResponse.redirect(new URL('/login', req.url))
  }

  try {
    await jwtVerify(token, new TextEncoder().encode(process.env.JWT_SECRET))
    return NextResponse.next()
  } catch {
    const response = NextResponse.redirect(new URL('/login', req.url))
    response.cookies.set(COOKIE_ACCESS_TOKEN_KEY, '', { maxAge: 0, path: '/' })
    return response
  }
}

export const config = {
  matcher: ['/dashboard/:path*', '/admin/:path*', '/mypage/:path*'],
}
```

middleware는 모든 요청 전에 실행되므로 최소한의 로직만. 로그인 페이지가 matcher에 포함되면 redirect loop가 생긴다.

## Refresh Token + RBAC

accessToken이 짧게 만료되는 구조라면 middleware에서 재발급까지 처리한다.

```tsx
export async function middleware(req: NextRequest) {
  const accessToken = req.cookies.get(ACCESS_COOKIE)?.value
  const refreshToken = req.cookies.get(REFRESH_COOKIE)?.value

  if (accessToken) {
    try {
      await verify(accessToken)
      return NextResponse.next()
    } catch {}
  }

  if (refreshToken) {
    try {
      const { payload } = await verify(refreshToken)
      const newAccessToken = generateNewAccessToken(payload)

      const response = NextResponse.next()
      response.cookies.set(ACCESS_COOKIE, newAccessToken, {
        httpOnly: true,
        secure: true,
        sameSite: 'none',
        path: '/',
        maxAge: 60 * 15,
      })
      return response
    } catch {}
  }

  return NextResponse.redirect(new URL('/login', req.url))
}
```

Role 기반 접근 제어(RBAC)도 middleware에서 처리한다:

```tsx
const ADMIN_PATHS = ['/admin']

const { payload } = await verify(token)
const userRole = payload.role as string

if (ADMIN_PATHS.some(path => req.nextUrl.pathname.startsWith(path)) && userRole !== 'admin') {
  return NextResponse.redirect(new URL('/unauthorized', req.url))
}
```

## 정리

| 기능 | 처리 방식 |
|---|---|
| 로그인 | accessToken + refreshToken 쿠키 저장 |
| accessToken 만료 | middleware에서 refreshToken 검증 후 재발급 |
| refreshToken 만료 | 로그인 페이지로 리디렉션 |
| SSR 인증 상태 | 쿠키 파싱 후 JWT 디코딩 |
| 관리자 페이지 보호 | role 검사 후 접근 제한 |

*쿠키는 여전히 유용한 도구다. 다만 사용처와 동작 범위를 정확히 이해해야 진짜 안전한 인증 시스템을 만들 수 있다.*
