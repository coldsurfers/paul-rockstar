# 카고 컬트 프로그래밍 — 직장인 개발자와 개발자의 차이

> "과연 개발 철학을 이해하지 못한 채, 직장인으로서의 개발만 하게 된다면 그 생애와 시장가치는 어디까지 이어질까?"

---

이미 레드오션이 된 개발자 시장에서 이 질문은 가볍지 않다.

다양한 개발자들과 협업하며 깨달은 것이 있다. 생각보다 많은 개발자들이 서로 다른 관점과 철학을 가지고 있다는 것. 그리고 그 차이가 결국 시장가치의 차이로 이어진다는 것.

---

## Cargo Cult Programming이란

**맥락과 철학을 이해하지 못한 채 무작정 따라 하는 행위**다.

유래는 2차 세계대전 이후 남태평양 부족에서 나왔다. 군용 비행기가 물자를 가져오는 걸 본 부족이 활주로를 본떠 만들었지만, 실제 보급은 오지 않았다. 형태만 흉내 냈을 뿐, 본질을 이해하지 못했기 때문이다.

코드도 마찬가지다. 이유 없이 프레임워크나 패턴을 차용하면 불필요한 복잡성과 유지보수 어려움만 남는다.

---

## 예시 ① TBD와 체리픽 배포

Trunk Based Development의 원칙은 단순하다.

- main에 머지되는 순간 즉시 배포 가능한 상태여야 한다
- main은 언제든 신뢰 가능한 브랜치여야 한다

하지만 실제 협업에서 이런 일이 벌어진다.

> "내 작업은 이미 main에 들어갔으니까, 체리픽해서 배포할게."

겉보기엔 틀린 말이 아니다. 하지만 이건 TBD를 표면만 흉내낸 전형적인 카고 컬트다.

- main은 배포 불가능한 브랜치가 되고
- 체리픽 배포가 일상화되며
- release와 main 간 형상 차이가 벌어지고
- TBD는 의미를 잃는다

맥락 없이 도입한 TBD는 생산성 향상은커녕 혼란만 낳는다.

## 예시 ② AbortController vs Path Based Blocking

JavaScript는 네트워크 요청 취소를 위한 표준 API를 제공한다.

```js
const controller = new AbortController()
fetch('/api/data', { signal: controller.signal })
  .catch(err => {
    if (err.name === 'AbortError') console.log('취소됨')
  })
controller.abort()
```

요청 제어 책임을 언어에 위임한다. 응집도와 재사용성이 유지된다.

이를 이해하지 못하면 이런 코드가 나온다.

```js
let activePath = null

async function fetchData(path) {
  if (activePath && activePath !== path) return // 실제로 cancel 되지 않음
  activePath = path
  const res = await fetch(path)
  return await res.json()
}
```

동작은 한다. 하지만 요청 제어 책임이 fetch 함수에 과도하게 붙고, 확장성이 없으며, 언어가 제공하는 표준 해법을 무시한다.

"동작만 하면 된다"는 사고가 낳은 카고 컬트다.

---

## 퀄리티의 정의

> "눈앞의 문제 해결에만 급급하지 않고, 미래까지 고려하며 지속가능한 방식을 선택하는 것"

카고 컬트에 익숙해진 개발자는 도메인 구현에만 멘탈 모델이 갇힌다. 프레임워크의 철학을 이해하지 못한 채 겉모습만 모방한다. 그 결과는 필연적으로 레거시 코드다.

---

본질을 이해하지 않고 겉만 흉내 낸 프로세스와 구현은 결국 레거시만 남긴다.

철학을 이해하고 가치를 추구하는 멘탈 모델을 갖출 때, 비로소 시장에서 오래 즐겁게 일할 수 있다.

*직장인 개발자로서의 생애는 이어갈 수 있을지 몰라도, 시장가치는 결국 줄어든다. 남는 건 레거시뿐이다.*
