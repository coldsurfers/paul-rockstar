# FSD와 성급한 추상화 — 본질을 찾아가는 과정

> 이해관계자가 많고 레거시가 쌓인 프로젝트. 겉은 FSD였지만 실제로는 중앙집중형 의존성이 높은 구조였다.

---

공통 컴포넌트를 개선하려는 A, 모든 이해관계자의 승인을 받고 머지하자는 B. 이 충돌은 단순한 프로세스 문제가 아니었다. 구조 자체가 만들어낸 병목이었다.

"넷플릭스 같은 빅테크 조직에서는 이런 문제를 어떻게 풀까?" — 그 고민이 FSD 관점으로 이어졌다.

---

## 1. 자명한 추상화 vs 성급한 추상화

레거시의 많은 문제는 성급한 추상화에서 비롯된다.

이 사례에서 A는 기존 shared 컴포넌트가 새 기획에 맞지 않자, HTTP 통신 라이브러리 사용에 제약을 두려 했다. 하지만 이는 문제와 직접적인 관련이 없는 제약이었고, 히스토리를 모르는 개발자가 제약을 간과하면 같은 문제가 반복된다.

함께 고민한 끝에 본질을 찾았다. 문제는 HTTP 통신 제약이 아니라 **권한 레이어 분리**였다.

```tsx
const APIPermissionLayer = ({
  children,
  fallback,
}: PropsWithChildren<{ fallback: ReactNode }>) => {
  const userRole = useUserRoleStore(state => state.user.role)

  if (userRole !== 'admin') {
    return fallback
  }

  return children // children에서만 http api fetch 수행
}
```

본질을 정확히 파악한 추상화는 불필요한 제약 없이 깔끔하다. 성급한 추상화는 유지보수 비용을 높이고, 장기적으로 조직 전체의 생산성을 저해한다.

---

## 2. 중복 허용 전략

FSD는 어느 정도 중복을 허용해 생산성을 높이는 구조다.

중앙집중형 구조에서는 하나의 페이지에서만 필요한 변경도 전사 공통 컴포넌트를 건드려야 하면 모든 팀의 동의가 필요하다. 승인 프로세스가 길어지는 이유다.

FSD에서는 과감히 중복을 허용한다.

- **공통화 비용 > 유지보수 이득**이면 중복 허용
- 각 슬라이스에 비슷한 DatePicker가 있어도 커스터마이징이 많으면 slice별로 유지
- 실험적인 컴포넌트는 UI Playground에서 검증 후 성공 시에만 공통화
- 실패하거나 특화된 건 slice에 그대로 둠

성급한 공통화를 지양하고, 기획 변화에 빠르게 대응할 수 있는 구조를 유지하는 것이 핵심이다.

---

## 3. 릴리즈 자유도 보장

모노레포 + FSD + MFE 환경에서 10명이 동시에 작업 중이라면, main에 머지된 모든 수정이 다음 배포자에게 부담이 된다. trunk와 release 브랜치 간 간극이 심화되고 기술 부채가 쌓인다.

이미 간극이 벌어진 상황이라면 **release 브랜치를 배포 기준**으로 삼는다.

```
main (trunk)
    ▲
    │ merge back
    ▼
release/X.Y  ← 이전 배포 태그에서 생성
    ▲
    │
hotfix / feature
```

1. 이전 배포 태그에서 release 브랜치 생성
2. 배포 대상 변경만 반영
3. 배포 후 main으로 병합
4. 반복

main이 불안정해도 안정 배포가 가능하고, trunk와 release 간 간극을 최소화할 수 있다.

---

## 마무리

FSD를 적용하며 깨달은 건 구조나 패턴 그 자체보다 **성급한 추상화를 피하고 자명한 추상화를 선택하는 판단력**이 훨씬 중요하다는 것이다.

때로는 과감히 중복을 허용하고, 배포의 자유도를 확보하는 것이 장기적인 생산성을 담보한다.

*AI가 수많은 대안을 제시해줄 수 있지만, 그중에서 본질적인 해결책을 가려내는 일은 결국 사람의 몫이다.*
