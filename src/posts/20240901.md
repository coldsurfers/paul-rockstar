# MySQL에서 위치 데이터를 다루는 법 — GeoHash로 최적화하기

> MySQL 기반 서비스에서 spatial data를 다루는 이야기. 다른 DB에서도 기본 골자는 참고할 수 있다.

---

## 왜 까다로운가

위치 데이터(longitude, latitude)는 다루기 까다롭다.

1. 소수점이 많다 — 정확한 파싱을 위해 전후처리 작업이 필요하다
2. 계산량이 많다 — row 수가 늘수록 계산량이 배가된다

---

## POINT 타입과 ST_Contains

MySQL은 공간 데이터를 위한 `POINT` 타입을 제공한다.

```sql
CREATE TABLE LOCATIONS (
  id INT AUTO_INCREMENT PRIMARY KEY,
  name VARCHAR(255),
  location_point POINT
);

SELECT
  ST_Y(location_point) AS latitude,
  ST_X(location_point) AS longitude
FROM LOCATIONS;
```

`ST_Contains`로 특정 바운더리 안에 포함된 좌표인지 쿼리할 수 있다.

```sql
SELECT
  NOT ST_Contains(
    (SELECT shape FROM spatial_data WHERE id = 1),  -- polygon
    (SELECT shape FROM spatial_data WHERE id = 2)   -- point
  ) AS is_outside;
```

문제는 연산 비용이다. 정확한 POINT 비교가 필요하고, Polygon이 복잡해질수록 연산량이 증가한다. Spatial index를 걸면 개선되지만, 서비스 관점에서 더 간단한 방법이 있다.

---

## GeoHash 도입

GeoHash는 좌표를 문자열로 인코딩하는 방식이다. 같은 prefix를 공유하는 geohash는 같은 지역 바운더리에 속한다.

ST_Contains 대신 `LIKE` 쿼리로 처리할 수 있다.

```sql
CREATE TABLE locations (
  id INT AUTO_INCREMENT PRIMARY KEY,
  name VARCHAR(255),
  latitude DOUBLE,
  longitude DOUBLE,
  geohash VARCHAR(12),
  INDEX (geohash)
);

-- 특정 바운더리 안의 데이터 조회
SELECT * FROM locations
WHERE geohash LIKE 'wydm%';
```

**ST_Contains vs GeoHash prefix 비교:**

| | ST_Contains | GeoHash prefix |
|---|---|---|
| 정확도 | 정밀 | 근사치 |
| 연산 비용 | 높음 (polygon 복잡도에 따라) | 낮음 (문자열 비교) |
| 인덱스 활용 | Spatial index | 일반 VARCHAR index |
| 쿼리 복잡도 | 높음 | 낮음 |

근사치 위치 필터링이 목적이라면 GeoHash가 훨씬 효율적이다.

## Prisma ORM + TypeScript 구현

**schema.prisma**

```prisma
model Location {
  id        Int    @id @default(autoincrement())
  name      String
  latitude  Float
  longitude Float
  geohash   String @db.VarChar(12)
  @@index([geohash])
}
```

**geohash 유틸**

```bash
yarn add ngeohash
```

```ts
import * as ngeohash from 'ngeohash'

export function encodeGeohash(lat: number, lng: number, precision = 6) {
  return ngeohash.encode(lat, lng, precision)
}
```

**데이터 삽입**

```ts
async function insertLocation(name: string, lat: number, lng: number) {
  const geohash = encodeGeohash(lat, lng)
  await prisma.location.create({
    data: { name, latitude: lat, longitude: lng, geohash }
  })
}
```

**prefix로 조회**

```ts
export async function findByGeohashPrefix(prefix: string) {
  return prisma.location.findMany({
    where: { geohash: { startsWith: prefix } }
  })
}
```

## 클러스터링 적용

지도 줌 레벨에 따라 geohash precision을 조절하면 자연스러운 클러스터링이 가능하다. 여러 prefix 그룹별로 데이터를 나눌 수 있다.

```ts
async function groupByGeohashPrefixes(prefixes: string[]) {
  const groups: Record<string, Location[]> = {}
  for (const prefix of prefixes) {
    groups[prefix] = await findByGeohashPrefix(prefix)
  }
  return groups
}
```

클러스터 중앙 좌표(centroid)가 필요하면:

```ts
function computeCentroid(locations: { latitude: number; longitude: number }[]) {
  const count = locations.length
  if (count === 0) return null
  return {
    lat: locations.reduce((sum, l) => sum + l.latitude, 0) / count,
    lng: locations.reduce((sum, l) => sum + l.longitude, 0) / count,
  }
}
```
